fedit / modality & shell — design explorer

Four axes are open: how modal the editor should be, what Esc does, what shell integration ships, and which prompt vocabulary we adopt. One decision is locked: Insert is the default mode — open a buffer, you can type. Everything below is a side-by-side comparison; the recommendation panel at the bottom names one pick per axis with a why. Decide; we plan the implementation in the next session.

01Modality scope

Four positions on the modal spectrum. Same task across all four (delete word, jump to line, find & edit) so the ergonomic delta is visible. Insert is the default in every option; Normal is reached via Esc.

02Esc behaviour

Three candidates. Four practical scenarios. Cells marked surprising are ones a typical user would not predict — the goal is to pick the model with the fewest of those.

Scenario A · "smart"Insert→Normal, Normal→opens prompt, prompt→prior focus B · pure Helix/vimInsert→Normal, Normal→nothing, prompt→close C · universal backcloses most-recent thing; Insert→Normal; never opens anything

S3 — Prompt opentext is theme green → closes prompt, restores prior focusIf you came from Editor/Insert, you land back in Insert. From Sidebar, back in Sidebar. → closes prompt, lands in NormalAlways Normal, regardless of where you came from. Consistent but loses context. → closes prompt, restores prior focusSame as A.
S4 — Sidebar focusedno prompt → Prompt opens, emptySame surprise as S2: Esc summoned something instead of escaping. → no-opYou must use Ctrl+E to leave the sidebar. → focus returns to Editor (Insert)"Back to where I was typing." Predictable.
How to read this. A has two surprising cells (S2, S4) — Esc opening something rather than closing it is the friction. B is consistent but offers no way to summon the prompt without learning the prefix keys (: /). C is the closest to "Esc means escape, every time" while still giving you a useful sidebar→editor return path.

03Shell integration scope

Goal: !ls -la runs and shows output; !!cargo test opens a buffer for the result; eventually !!htop works as a real interactive terminal. Each rung delivers most of the previous one's value with progressively more infrastructure.

MVP-1 · async !cmd to dock effort · ~2 days

Ships

  • Prompt prefix ! spawns a subprocess via Process.Start (reuse the clipboardCopy pattern in Runtime.fs:76-109).
  • Runs async via Task.Run; output posted back as Msgs through the existing ConcurrentQueue.
  • New DockOutput variant rendered by View.fs alongside DockCompletions / DockInfo.
  • Ctrl+C while focused on the prompt kills the running command.

Doesn't ship

  • No !! (buffer output).
  • No ANSI color rendering — escape codes shown verbatim or stripped.
  • No interactivity. Output-only.

MVP-2 · + read-only output buffer for !!cmd effort · +1–2 days (total ~4 days)

Ships

  • Prompt prefix !! opens a new buffer marked ReadOnly (new flag on BufferState).
  • Output streams into the buffer as it arrives; cursor follows the bottom unless the user scrolls up.
  • Buffer name is the command (!!cargo test); closes via :quit like any other buffer.

Doesn't ship

  • Still no interactivity. The process can't be typed at.
  • Still no ANSI color (rung 3).
  • Read-only means Ctrl+C in the buffer doesn't send SIGINT — kill only from the prompt.

MVP-3 · + ANSI SGR colour effort · +2–3 days (total ~1 week)

Ships

  • A small SGR parser converts \x1b[31m etc. into styled cells. We already render styled cells — this is wiring, not new infra.
  • Works in both dock output and !! buffers. cargo, grep --color, git diff, ls -G all look right.

Doesn't ship

  • Cursor-motion escapes (CUP, ED, CUU/CUD/CUF/CUB) are ignored. Plain output is fine; full-screen TUIs like htop/vim still scramble.
  • No mouse passthrough.

Full · interactive PTY buffer effort · 2–4 weeks, possibly more

Ships

  • !!cmd opens a buffer backed by a real PTY (openpty on macOS/Linux via P/Invoke; Pty.Net on Windows).
  • Keystrokes in the buffer are forwarded to the subprocess; ANSI cursor motion is parsed and applied to a cell grid; resize emits SIGWINCH.
  • vim, htop, fzf, nvim, an interactive REPL all run inside fedit.

What it costs

  • A partial terminal emulator inside fedit — biggest test surface in the codebase.
  • New failure modes: dead subprocess, broken pipe, child grandchildren, signal forwarding bugs.
  • Maintenance forever, because every new TUI program is a new regression case.

Reality check. Full PTY is the largest single feature fedit would ever add. The codebase has zero PTY infrastructure today — the only existing Process.Start usage is fire-and-forget clipboard copy/paste in Runtime.fs:76-109. Recommend shipping MVP-1 + MVP-2 + MVP-3 in that order and revisiting Full only once we have a concrete use case beyond "wouldn't it be cool if vim ran inside fedit." MVP-3 already covers cargo, git, grep, just, dotnet — the actual day-to-day commands.

04Recommendation

One pick per axis. Not the only valid combination — but the one that lands the most value for the least risk, and lets us learn from real use before committing to bigger investments.

What I'd build first

Modality
Helix-lite
Full Helix is a multi-month commitment whose payoff (selection-first grammar, multi-cursor) we can't justify before we know how users hold the editor. Modal-launcher is barely modal — Esc just opens a prompt. Modeless+unified is the safest skip-this-round option but gives up the "modal feels right" thing the user explicitly wants. Helix-lite gets the feel for ~1 week of work and leaves a clean upgrade path to Full Helix later if appetite grows.
Esc
Universal back-button (Option C)
Option A's "Esc opens the prompt" cells (S2, S4) are the friction; every user instinct says Esc closes things. Option B is consistent but offers no Esc-driven path out of the sidebar. Option C is the closest to "Esc means escape, every time" while still giving the sidebar→editor return that the user wants. The prompt gets opened by typing : / / / ! from Normal mode, which is also the Helix muscle memory.
Shell
MVP-1 → MVP-2 → MVP-3, in that order. Defer Full.
MVP-1 ships in ~2 days and covers !ls, !grep, !just check. MVP-2 (+1–2 days) adds the captured-output buffer for longer commands like !!cargo test. MVP-3 (+2–3 days) adds SGR colour so output stops looking dead. Total: ~1 week for the 90% case. Full PTY (2–4+ weeks) only makes sense once we know which interactive TUI we actually need inside fedit — and the answer might be "none, just keep using the host terminal."
Ship order if all three accepted. Modeless prompt unification first (already specced — Direction A in TODO.md), since both modality and shell features land on top of it. Then MVP-1 shell. Then Helix-lite modal layer with Esc-as-back. Then MVP-2 + MVP-3 shell in either order. Each step is independently useful — bail at any rung without leaving the editor in an awkward middle state.
